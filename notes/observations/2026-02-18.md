# Observations — 2026-02-18

## ARIA Causal Trace — Distributed Acronym Memory vs. Localized Proper Noun Memory

**Tool:** ROME causal tracing (Meng et al. 2022), running on GPT-2 XL via Apple Silicon (MPS)
**Prompt:** "ARIA stands for Accessible Rich Internet" → target: " Applications"
**Baseline:** "The Space Needle is in the city of" → target: " Seattle"

**Observation:** ARIA and Space Needle produce structurally different causal trace signatures, suggesting fundamentally different memory storage mechanisms.

**Space Needle (proper noun fact):**
- Tight, localized signal at the last subject token ("le*") in mid layers
- Clean band, single token, single layer range
- Consistent with ROME's core finding: factual associations are stored in specific MLP layers at the subject's last token

**ARIA (acronym expansion fact):**
- Early layers: diffuse signal across acronym tokens AR* and IA*
- AR* and IA* show strong MLP signal in layers 0-15 — acronym recognition happening early
- "Internet" row: carries strong MLP signal from layer ~28 through layer 47 in both the state restoration and MLP-specific heatmaps
- No single clean localized band — signal is distributed across the full expansion context

**Interpretation:** ARIA is not stored as "ARIA" → "Applications." It is stored as a completion pattern requiring the full expanded context — "Accessible Rich Internet" → "Applications." The key-value pair the MLP is retrieving is anchored to "Internet" as the final content word of the expansion, not to the acronym token itself.

This is a structurally different memory type than proper noun facts:
- **Proper noun facts:** Single subject token → localized mid-layer MLP retrieval
- **Acronym expansion facts:** Distributed across expansion tokens → late-layer MLP retrieval anchored to final content token

**Connection to prior work:** Consistent with the logit lens findings showing ARIA requiring full context to resolve (Explorer/Protocol oscillation until layer 35). The causal trace explains WHY — the model needs the full "Accessible Rich Internet" context to fire the correct MLP memory. The acronym alone is insufficient.

**Broader implication:** Acronym-based domain knowledge may be systematically harder to localize and edit via ROME-style interventions because the "subject" is not a single token but a distributed multi-token expansion. This has implications for model editing in specialized domains where knowledge is encoded in acronyms (medical, legal, accessibility).

**Needs:** Run causal trace on WCAG, Paris/Berlin (proper nouns as controls), and alt text to build out the taxonomy of memory types.

## Alt Text Probe — Knowledge Encoding vs. Retrieval Context

**Prompts tested:**
- "Alt text describes an" → target: " image"
- "In HTML, the alt attribute provides a text description of an" → target: " image"

**Observation:** The two prompts produce dramatically different retrieval outcomes despite testing identical knowledge:

| Model | Prompt | First rank 1 | Final prob |
|-------|--------|-------------|------------|
| Medium | alt text | never | stuck ~rank 6 |
| Large | alt text | layer 34/36 | ~9% |
| XL | alt text | layer 42 | 32% |
| XL | HTML/technical | never | 11% (rank 2) |

The more specific HTML framing performs *worse* than the ambiguous "alt text" prompt for XL. The HTML context activates "element" as a dominant prior (HTML element, DOM element, form element), which overwhelms the accessibility retrieval signal. The "alt text" framing, despite the alt/alternate ambiguity, primes the accessibility context more directly.

**Medium failure pattern:** MLP peak ~35 logits for "image" but model outputs "experiment" — massive internal signal that cannot overcome the syntactic prior for "describes an [X]" completions.

**XL capacity ambiguity:** XL entertains "alternate" (layers 29-33) before resolving to "image" — correctly recognizing that "alt" is ambiguous between "alternate text" and the HTML attribute. More domain knowledge creates more competition before resolution.

**General principle — Specialized Domain Retrieval:** Any specialized domain vocabulary sits at the intersection of multiple strong general frames. Accessibility concepts compete with HTML/web dev, medical terminology competes with general anatomy, security concepts compete with general computing vocabulary. The model may encode domain knowledge in MLPs but retrieval depends critically on which frame the prompt activates. Knowing and being able to retrieve are not the same thing.

**Implication for probe design:** Prompt construction is a methodological contribution in itself. Probes must be designed to activate the target domain frame without triggering stronger competing priors. This is a general problem for interpretability work on specialized domains, not just accessibility.

**Needs:** Run Medium and Large on the HTML prompt for comparison. Test additional accessibility prompts that avoid both the alt/alternate ambiguity and the HTML/element frame to establish a cleaner retrieval baseline.
